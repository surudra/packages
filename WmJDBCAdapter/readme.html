2.0 Known Issues

2.1 Adapter Issues

- During configuration of the StoredProcedureWithSignature service, the value of the Output Field changes if you change an IN expression from '?' to a 'fixed value', or vice versa.  To resolve this, after making changes to the IN expression field, select or type the Output Field name again.

- For DB2 for OS/390, non repeatable read does not function when the transaction isolation level is set to 2. For Teradata, non repeatable read cannot be avoided even after setting the transaction isolation level to 8.

- For Informix, phantom read does not function when the transaction isolation level is set to 4. For Teradata, phantom read cannot be avoided even after setting the transaction isolation level to 8.

- For Oracle, when using Oracle Array or Struct as OUT parameters in an Oracle stored procedure, use the StoredProcedureWithSignature service. StoredProcedure service currently does not support Oracle procedures with a similar signature.

- For Teradata, dirty read does not function when the transaction isolation level is set to 1.

- For Teradata, the table type is shown as ‘T’ instead of ‘Table’. This is because the driver returns ‘T’ as the type for the table, during table name look up in service templates.

- For Teradata, the ExecuteService adapter service does not honor the transaction boundaries laid down by ART.

- For Oracle 10g, StoredProcedureWithSignature adapter service does not work with Oracle functions having an Array type in its signature.

- To avoid a file conflict when using Informix, you must disable the WmTomcat package in the Integration Server.  Be aware that this also disables support for JSPs.

- When using the BatchInsert or BatchUpdate service with Sybase, you may receive the following error message, if the service attempts to return a large number of records:

"JZ0BE:BatchUPdateException:Error occurred while executing batch statement:Too many parameters--symbol table overflow"

To resolve this problem, apply a jConnect 5.5 emergency bug fix EBF 11473, which is available at: http://downloads.sybase.com/swd/swx/sdfile.stm.

- You cannot enable C-JDBC 3.2 driver connections using JVM 1.4.1_01, 1.4.1_03, or 1.4.1_05.

StoredProcedureWithSignature services and StoredProcedureNotificationWithSignature notifications do not work with Stored Procedures containing a parameter of type Oracle Cursor, when the ref cursor is declared as a cursor type that is defined as a strong type with the %ROWTYPE attribute. 

Instead, define the ref cursor as a weak type.

Type 3 Net

StoredProcedureWithSignature services and StoredProcedureNotificationWithSignature notifications are supported only on DB2 for OS/390 V6.

6.0 Globalization

webMethods JDBC Adapter 6.5 conforms to the internationalization standards of webMethods and includes support for operation in any country, locale, or language as specified in the webMethods Installation Guide. It was tested with non-English configurations and non-ASCII data.

6.1 Configuration

If you expect to display non-English data in your webMethods tools, make sure an appropriate font is installed in your operating environment. If you see hollow boxes or black squares where you expect to see non-English characters, modify your JRE font.properties file to reference an appropriate font. For information on modifying your font.properties file, go to Sun's Java website. For instructions, see:

http://java.sun.com/j2se/1.4.2/docs/guide/intl/fontprop.html

http://java.sun.com/j2se/1.5.0/docs/guide/intl/fontconfig.html

http://www.unicode.org/help/display_problems.html

The configuration of the database you use with webMethods JDBC Adapter 6.5 affects the characters the database can contain. If you select a non-Unicode encoding for the database, characters from outside that encoding will be lost when you store data in the database. webMethods recommends that you use the most appropriate Unicode encoding for your database character set. See the webMethods Installation Guide and your database vendor's documentation for instructions on choosing and setting the character encoding for your database and environment.

6.2 Localization

webMethods JDBC Adapter 6.5 is provided with an English user interface, messages, and documentation. Language packs containing support for other languages, when available, can be installed at any time using the webMethods Installer. webMethods JDBC Adapter 6.5 supports simultaneous multi-locale operation.

7.0 Core Security Facilities

Effective with webMethods 7, all webMethods products address seven core security facilities, either by providing all of them or explaining what aspects are not met. Additional information can be found on Advantage in https://advantage.webmethods.com/product_security_information.

webMethods JDBC Adapter does not conform to all webMethods core security facilities requirements as described in the following paragraphs.

7.1 Authenticated Network Traffic

webMethods JDBC Adapter is capable of authenticating all network traffic. Instructions on configuring webMethods JDBC Adapter to perform authentication can be found in the webMethods JDBC Adapter User’s Guide (see section 8.0 Documentation in this readme).

7.2 Encrypted Network Traffic

webMethods JDBC Adapter is not capable of encrypting any network traffic.

webMethods suggests using a Virtual Private Network (VPN) as an alternative to encryption.

7.3 Inbound Password Protection

webMethods JDBC Adapter does not include any inbound passwords, so this requirement is not applicable.

7.4 Outbound Password Protection

webMethods JDBC Adapter stores all outbound passwords in encrypted form, which are resistant to tampering or theft even if they are stolen.

7.5 Private Key Protection

webMethods JDBC Adapter does not include any private keys, so this requirement is not applicable.

7.6 Security Auditing

webMethods JDBC Adapter relies on the hosting Integration Server to provide security auditing. See the corresponding section of the Integration Server readme which provides directions on Integration Server security auditing.

7.7 Hardwired Passwords

webMethods JDBC Adapter does not contain any hardwired (unchangeable) passwords.
