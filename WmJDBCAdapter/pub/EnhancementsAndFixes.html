1.0 Fixes and Service Packs Included in This Release

webMethods JDBC Adapter 6.5 includes the following fixes and service packs that were shipped since the last release.

- JDBC_6-0-3_Fix1 through JDBC_6-0-3_Fix18 on the JDBC Adapter 6.0.3

- JDBC_6-0-3_FP1_Fix1 through JDBC_6-0-3_FP1_Fix14 on the JDBC Adapter 6.0.3 FP1

For detailed information on these fixes and feature packs for webMethods JDBC Adapter, see the Knowledge Base on the webMethods Advantage Web site.

2.0 Enhancements

webMethods JDBC Adapter 6.5 includes the following enhancements:

Polling Notification Enhancements

-  Improve the polling notification migration by providing better error handling messages when migrating in a non-Suspended state.

-  Clean up the database objects for a given adapter notification using a utility service.

-  Enable a Basic Notification to not delete processed records in the notification's associated table.

-  Retrieve the old, new, and both old and new values for an Update Notification.

-  Include a WHERE clause in a Basic Notification.

-  Enable the JDBC Adapter to obtain the database object information for adapter notifications using a utility service.

-  Allow a polling notification to publish locally to the webMethods Integration Server.

Connection Management Enhancements

-  Support the setting of DataSource property that use ; (semicolons) as delimiters for multiple key value pairs.

-  Update the password of an existing connection programmatically, using a public service.

-  Specify the user name and password credentials at run time for JDBC Adapter connections.

-  Configure the JDBC Adapter connections automatically, using a public service.

-  Set the transaction isolation levels at the JDBC Adapter connection level.

Oracle-specific Enhancements

-  Provide support for Oracle-specific CLOB and BLOB data types as a parameter input type, in JDBC Adapter services.

-  Support the Oracle ARRAY and STRUCT data type objects as input and output parameters in the adapter services.

-  Support streaming with the JDBC Adapter.

Other Notable Enhancements

-  Allow a java or a flow service obtain a connection from the JDBC Adapter connection pool using the new adapter service - ExecuteService.

-  Expand the UpdateServiceSchema service for use with StoredProcedure and StoredProcedureWithSignature adapter services.

-  Provide a new dummy string data type SetAsString in the adapter services. You can set the value of this dummy data type by passing a value of java.lang.string data type. When selecting the SetAsString data type, the JDBC Adapter treats the data as a string and does not convert it to the underlying database column type or to the equivalent JDBC type.

-  Expose the resulting number of rows for a SelectSQL adapter service and for the Select query in CustomSQL and DynamicSQL adapter services.

-  Execute the CustomSQL and DynamicSQL services for a Select query without explicitly specifying the output parameters in the adapter service template.

-  Provide a new data type, java.lang.Object which uses the setObject method for all supported JDBC data types.

-  Add the Query Timeout parameter for adapter services and notifications.

-  Suppress the error logging when the database is unreachable or down. The suppressed error is logged for five consecutive times and then suppressed.

3.0 Resolved Issues

The issues listed below have been addressed in this release.

1-157LXR (JDBC_6-0-3_FP1_Fix14 and JDBC_6-0-3_Fix18)
While executing the StoredProcedure service, an error with message "Maximum cursor count(50) exceeded" is thrown.
This error is thrown when the StoredProcedure service that returns multiple result sets is executed continuously in a loop. This happens because the Result Set objects do not get closed.

1-TLKDI (JDBC_6-0-3_FP1_Fix14 and JDBC_6-0-3_Fix18)
Errors occur at Integration Server restart while loading packages that contain disabled connections and notifications.
When the Integration Server loads a package containing disabled JDBC Adapter connections and notifications, the following errors are thrown:
[ART.0114.1007E] Adapter Runtime: Error Logged. See Error log for details. Error: [ADA.1.311] The connection is not available for " initCallBack() ".
[ART.0116.3038E] Adapter Runtime (Notification): Error in Notification Callback:initCallback notification JDBCTest:insertNotification.
This error occurs because the Adapter Runtime (WmART) requires the connection even though it is disabled.

1-15RO9P (JDBC_6-0-3_Fix18)
An error "No method createSchemaList in class pub.jdbcAdapter" is thrown, after installing the JDBC_6-0-3_Fix17.
This error occurs because the WmJDBCAdapter package is not loaded completely during installation.

1-15E7Z4 (JDBC_6-0-3_FP1_Fix13)
In webMethods JDBC Adapter 6.0.3, when the StoredProcedureWithSignature service is used to invoke a StoredProcedure with a CLOB or a BLOB data type as output, an invalid column type error is thrown.
When a StoredProcedureWithSignature service invokes a StoredProcedure service which has a CLOB or a BLOB data type as output data type, then the adapter sets the CLOB or BLOB data type as 'Other' data type. This happens because Oracle returns 'Other' as the metadata for the CLOB and the BLOB data type. Due to this, an invalid column type error is thrown.

1-14L10N (JDBC_6-0-3_FP1_Fix13)
In webMethods JDBC Adapter 6.0.3, the Select service on Oracle databases that has the selected column data type as CHAR(n) is handled incorrectly.
In webMethods JDBC Adapter 6.0.3, for the Select service if the selected Oracle database Column Type is CHAR(n), the corresponding JDBC Type is taken as CHAR. Due to this, the service does not work properly.
This issue is now resolved by supporting another JDBC data type 'ORACLEFIXED_CHAR'. Now, in the Select tab of the Select service, if the Column Type is CHAR(n), 'ORACLEFIXED_CHAR' is automatically populated in the JDBC Type field. But, in the WHERE tab, you need to select the corresponding data type in the JDBC Type field.

1-14S6AH (JDBC_6-0-3_FP1_Fix12)
In webMethods JDBC Adapter 6.0.3.1, no exception is thrown when the commit of a local transaction fails.
In webMethods JDBC Adapter 6.0.3.1, when the commit of a local transaction fails, the adapter logs the exception, and neglects it. It does not throw an exception.

1-14JW5P (JDBC_6-0-3_FP1_Fix12 and JDBC_6-0-3_Fix17)
For an UpdateNotification, when all the columns are disabled for notification, an error message is not displayed.
In webMethods JDBC Adapter, when using the UpdateNotification, at least one of the columns of the database table has to be enabled for notification. If all the columns are disabled for notification, an error message should be displayed.
This issue is now resolved. If all the columns in the table are disabled for notification, an error message is displayed on the Integration Server console, and the UpdateNotification cannot be enabled.

1-145RNX (JDBC_6-0-3_FP1_Fix12) (JDBC_6-0-3_Fix17)
In webMethods JDBC Adapter, the StoredProcedure adapter service has a limitation of returning only a maximum number of 20 result sets.
The StoredProcedure service in the webMethods JDBC Adapter can be configured to return multiple result sets. By default, the service can return only a maximum of 20 result sets. A new result set index cannot be added manually.

This issue is now resolved. A new watt property is introduced, watt.adapter.JDBC.StoredProcedureMaxCursorIndex, that enables you to specify the required maximum number of result set index.
To specify the required maximum number of result set index:

1. From the Integration Server Administrator, click Settings > Extended. 

2. Click Edit Extended Settings.

In the Extended Settings editor, type the following:

watt.adapter.JDBC.StoredProcedureMaxCursorIndex= <<no of result sets required>>

  where <<no of result sets required>> is the required maximum number of result sets. For example, to return a maximum number of 30 result sets from the StoredProcedure service, set the property as:

watt.adapter.JDBC.StoredProcedureMaxCursorIndex=30

Note: The parameter name is case sensitive. The value of the parameter should be greater than zero but has no upper limits. However, it is better to provide a value within the practical limits. A large value can lead to an unpredictable behavior of the system. Instead, you can configure another StoredProcedure service for the next set of result sets.

3. Click Save Changes. A key for the new property appears in the Extended Settings list.

4. Click Show and Hide Keys. You will see "watt.adapter.JDBC.StoredProcedureMaxCursorIndex" listed as a Key and is visible.

5. Restart the webMethods Integration Server for the changes to take effect.

For more information about working with extended configuration settings, see the webMethods Integration Server Administrator's Guide.

1-12XS9D (JDBC_6-0-3_FP1_Fix12 and JDBC_6-0-3_Fix17)
While configuring JDBC services and notifications, the default value is set incorrectly in the Maximum Row field.
In SelectSQL, CustomSQL, DynamicSQL services, and Insert, Update, Delete notifications, the default value is set incorrectly in the Maximum Row field. By default, the maximum value should be set as 0 which retrieves all the records from the database.

1-1070SQ, 1-11FV6H (JDBC_6-0-3_FP1_Fix11 and JDBC_6-0-3_Fix16)
JDBC Adapter notifications submit the same document twice.
Inserting the same row/identical data in the database more than once results in duplicate notifications even if Select 'DISTINCT' is specified at design time. This situation occurs for any Insert/Update/Delete notification and results in an error for the second document published.
This issue is now resolved. The JDBC Adapter now suppresses the duplicate notification if 'Select DISTINCT' is specified at design time.

1-12ATAF (JDBC_6-0-3_FP1_Fix11 and JDBC_6-0-3_Fix16)
JDBC Adapter that uses the SQL Server 2005 JDBC Driver incorrectly displays the stored procedure name and cannot execute that procedure.
Stored Procedure service names are incorrectly displayed as being enclosed in quotation marks. For example, the Stored Procedure service named sp_GetEmployee;1 is displayed as "sp_GetAllEmployee;1". At service run time, execution fails with the following messages:

Could not find stored procedure 'Test.dbo.sp_GetAllEmployee;1'."

Could not find stored procedure 'Test.dbo.sp_GetAllEmployee;1'.

This issue has been resolved. Stored Procedure service names now appear correctly, and the services run successfully.

1-12GCBP (JDBC_6-0-3_FP1_Fix11 and JDBC_6-0-3_Fix16)
Exception thrown when creating a Stored Procedure With Signature Service that uses specific MS SQL Server 2005 data types.
When creating a Stored Procedure With Signature Service with parameters whose data types are nchar, ntext, nvarchar, nvarchar(MAX), uniqueidentifier, and xml, the following exception is thrown at execution:
com.wm.app.b2b.server.ServiceException: [ART.114.205] Adapter Runtime (Metadata): Unable to lookup the resource domain values.null on procedureSignature.colIndexes [Ljava.lang.String;@7e9c7d java.lang.Exception: Adapter values are not available.
This issue has been resolved. The JDBC Adapter now includes support for nchar, ntext, nvarchar, nvarchar(MAX), uniqueidentifier, and xml data types in MS SQL Server 2005.

1-12AT6N (JDBC_6-0-3_FP1_Fix11 and JDBC_6-0-3_Fix16)
The JDBC Adapter does not correctly handle certain MS SQL Server 2005 data types, causing notifications to fail.
Using the JDBC Adapter with the MS SQL Server 2005 driver, the driver returns the columnSize in the metadata for the data types smallmoney, timestamp and uniqueidentifier. (These data types are displayed as smallmoney(10,4), timestamp(8), and uniqueidentifier(36), respectively, in the Developer.) The adapter then uses the metadata, including the column size, when it creates the buffer table when the notification is enabled. However, because MS SQL Server 2005 does not allow tables with these columns to be defined with the column size, enabling the notification fails.
This issue has been resolved. The JDBC Adapter now successfully enables notifications for tables that have columns for data types smallmoney, uniqueidentifier, and timestamp.

1-12ASER (JDBC_6-0-3_FP1_Fix11 and JDBC_6-0-3_Fix16)

When enabling an ordered notification for the JDBC Adapter on Informix, the adapter throws the following error: [ART.116.3038] Adapter Runtime (Notification): Error in Notification  Callback:enableCallback notification Bossini.test: INot.[ADA.1.316] Cannot execute the SQL statement "sql_statement".
The "sql_statement" specified indicates that the SQL statement was trying to create a procedure for a table in which "NOT NULL" is explicitly mentioned in the column descriptions. This error occurs because the stored procedure that is called for enabling an Informix ordered notification generates a syntax error if a column description has a "NOT NULL" clause.

1-111D0X, 1-11FCXF (JDBC_6-0-3_FP1_Fix10 and JDBC_6-0-3_Fix15)
BasicNotification fails if the database is down during Integration Server startup.
After restarting the Integration Server with the database down, running a BasicNotification after the database is brought back up throws the following error:
<notification-name>[ART.116.3001] Adapter Runtime (Notification): Notification error during execution of <notification-name>. Error: [ADA.1.316] Cannot execute the SQL statement "null". "(17104) SQL statement to execute cannot be empty or null".
This condition occurs because the BasicNotification does not initialize after the database comes back up, resulting in the Select query being set to null.
This issue is now resolved, and the BasicNotification is properly re-initialized.

1-ZBTJR (JDBC_6-0-3_FP1_Fix9)
When the webMethods Integration Server is shut down during the execution of a flow service, the JDBC connection is closed and open transactions are not explicitly rolled back by ART. As a result, the transaction is left open when the connection is closed, and the default behavior of the connection resource is applied to the open transaction. With an Oracle JDBC driver, the transaction is committed instead of being rolled back.
Now the JDBC Adapter explicitly rolls back the open transaction instead of committing it.

1-ZW3SN (JDBC_6-0-3_FP1_Fix9)
Buffer table created with incorrect NCHAR column size prevents the corresponding notification from being enabled.
When a JDBC Adapter notification is configured with the base table containing an NCHAR column, the notification cannot be enabled because the JDBC Adapter creates a buffer table with the incorrect NCHAR column size.
Now, the JDBC Adapter creates a buffer table with the correct NCHAR column size.

1-Y1X74 (JDBC_6-0-3_FP1_Fix9)
In the JDBC Adapter, an OrderedNotification throws an error if the column has the Identity set.
In a table where one column has "Identity" set as "yes", creating an ordered notification on that table causes an adapter runtime error to be thrown. This error condition is due to a syntax error in the generated trigger code.
Now, the JDBC Adapter ordered notification creates a trigger with the correct syntax.

1-XB7IH, 1-ZVYR1 (JDBC_6-0-3_FP1_Fix8 and JDBC_6-0-3_Fix14)
Notifications do not work after the database is down during Integration Server (IS) startup.
If the database is down during IS startup, the connection pool is disabled. For enabled notifications, you will see messages that indicate there is no connection available from the pool.
Even after the database comes back up and the connection pool is enabled, the insert/update/delete notification functions still do not work and the notification throws the following error:
[ADA.1.327] This notification is not ready to be enabled.
This fix allows enabled notifications to continue to retrieve data from existing buffer tables after re-establishing the connection pool.

1-WX8E9 (JDBC_6-0-3_FP1_Fix7 and JDBC_6-0-3_Fix13)
IS trigger retry with JDBC Adapter service in Integration Server 6.5 environment fails.
With WmART 6.1 and earlier, if the database is down, WmART destroys all connections in the pool and an IS trigger tries to execute the adapter service. During the retry, WmART passes a connection from the connection pool to the adapter service and, since all connections were previously destroyed, WmART and the JDBC Adapter attempt to make new connections. If the attempt fails, the IS trigger continues to retry executing the adapter service. The retry attempts continue until the configured number of IS trigger retries are exhausted or until the connections are created and the adapter service is executed successfully.
With WmART 6.5, if the database is down, the IS trigger tries to execute the adapter service, but makes only one attempt. During the retry, if the connections cannot be created, the IS trigger stops and the adapter service fails to execute.

The fix addresses this issue by:

- Adding a new extended property named

  watt.adapter.JDBC.database.fatalErrors=+<errorcode>

- Adding new fatal error codes that indicate the database is down to the adapter's list of fatal error codes. Those error codes are:

08S01    17410

28       17416

1012     40003

1033     -601

1089     -30081

1092     -79716

3114     -99999

12571    HY000

17001    JZ0C0

17002

This list is a combination of common fatal error codes from different databases.

With Integration Server 6.5, if the JDBC Driver returns one of the error codes in the list, the IS trigger will retry executing the adapter service for the configured number of IS trigger retries. If the IS trigger does not retry the adapter service when the database is down, it is most likely due to an error code being received that is not in the list. In this case, you can add the missing error code to the list using the following procedure:

1. Locate the error code returned when the database is down:

a. Change the IS logging level to 8.

b. In the server log file, search for adapter message

      ADA.0001.0113.

      For example, the adapter message below shows the error code ORA-12535:

[ADA.0001.0113V4] Received SQLException:

"(72000/12535) ORA-12535: TNS:operation timed out".

2. Add the missing error code to error code list:

a. From the Integration Server Administrator, click "Settings">"Extended".

b. Click "Edit Extended Settings". In the Extended Settings editor, type:

   watt.adapter.JDBC.database.fatalErrors=+<errorcode>

     where <errorcode> is the error code you are adding.

Using the example above, to add error code ORA-12535 you would type:

watt.adapter.JDBC.database.fatalErrors=+12535

c. Click "Save Changes".

d. Restart the webMethods Integration Server for the changes to take effect.

For more information about working with extended configuration settings, see the webMethods Integration Server Administrator's Guide.

NOTE: The new property is not database-driver specific, but instead applies to all database drivers. When using the JDBC Adapter with Integration Server 6.5, if you previously customized fatal errors using the property:

watt.adapter.JDBC.<database_driver>.fatalErrors=

(as described in the webMethods JDBC Adapter User's Guide), those customizations will still be recognized. However, we recommend that you use the new property in the future to add fatal error codes.

1-OTGQB (JDBC_6-0-3_Fix2)
This fix provides a new flow service called pub.jdbcAdapter.UpdateNotificationSchema that changes schema names configured in the following types of notifications: InsertNotification, DeleteNotification, UpdateNotification, OrderedNotification, and BasicNotification. The UpdateNotificationSchema service is located in the pub.jdbcAdapter folder in WmJDBCAdapter package.
The service always validates the new schemas against the notification settings, and sets the state of the notification to the state it is in before it is updated, regardless of whether the update is successful.

The flow does the following:

1. The flow checks the current status of the notification.

2. If the notification is enabled, the flow checks the forceDisable parameter. If forceDisable is false, the flow reports the error. If forceDisable is true, the flow disables the notification.

3. The flow validates the schema against the notification's settings. The validation requires that the schema be in the same catalog that is configured with the notification and that the table, synonym, view, or alias, configured in the notification be in the schema. If the schema is not valid, the flow throws an exception.

4. The flow updates the notification property with the new schemas, and throws an exception if the input fields for the new schemas are not valid.

5. If the original notification state is enabled, the flow enables the notification.

The input signature for the service is as follows:

- nodeName is a required String field. It sets the name of the notification to be updated.

- forceDisable is a required Boolean field. If the value of this field is set to true, the service disables the notification that is in the enabled state.

- allSchemaChange is an optional Record field. It is used if the update is to occur on every schema name in the notification.

  - useCurrentSchema is an optional Boolean field. It sets all schema names to <current schema>

  - schemaName is an optional String field. It sets all schema names to the value of this field.

- schemaChanges is an optional record list field. It is used to make individual schema changes, by searching for any schema name set in the existingSchema field and then replacing it withthe corresponding schema name set in the newSchema field.

  - existingSchema is a required Record field. It identifies the schema name to change

    - useCurrentSchema is an optional Boolean field. It identifies the <current schema>.

    - schemaName is an optional String field. It sets the schema to be replaced.

  - newSchema is a required Record field. It defines the new Name.

    - useCurrentSchema is an optional Boolean field. It sets new schema names to <current schema>

    - schemaName is an optional String field. It sets new schema names to this name.

Keep the following points in mind when configuring the pub.jdbcAdapter.UpdateNotificationSchema service:

-  The allSchemaChange and schemaChanges fields are mutually exclusive. This means that if allSchemaChange is set, then schemaChanges will be ignored.

-  The useCurrentSchema and schemaName fields are mutually exclusive everywhere they occur in the input. This means that If useCurrentSchema is set to true, then the value in schemaName will be ignored.

-  A localized string is used for the <current schema>.

-  The service doesn't generate output. It throws AdapterServiceException wrapping exceptions from ART and JDBC Adapter.

The following section lists the input fields to set for certain use cases. Fields that do not list an input value should be left empty.

To set all schemas to the <current schema>:

  - nodeName           folder1:notification1

  - forceDisable        true

  - allSchemaChange

    - useCurrentSchematrue

To set all schemas to schema A:

  - nodeName           folder1:notification1

  - forceDisable        true

  - allSchemaChange

    - schemaName       A

To change occurrences of the <current schema> to schema A:

  - nodeName           folder1:notification1

  - forceDisable        true

  - schemaChanges

   - schemaChanges[0]

     - existingSchema

       - useCurrentSchema  true

     - newSchema

       - schemName          A

To change occurrences of schema A to the <current schema>:

  - nodeName           folder1:notification1

  - forceDisable        true

  - schemaChanges

    - schemaChanges[0]

      - existingSchema

        - schemaName       A

      - newSchema

        - useCurrentSchema  true

To change occurrences of schema A to schema A1, and occurrences of schema B to the <current schema>:

  - nodeName           folder1:notification1

  - forceDisable        true

  - schemaChanges

    - schemaChanges[0]

      - existingSchema

        - schemaName    A

      -newSchema

        - schemaName    A1

    - schemaChanges[1]

      - existingSchema

        - schemaName      B

      - newSchema

        - useCurrentSchema true
